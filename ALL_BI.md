# Глава 1. РАЗВЕРТЫВАНИЕ СРЕДЫ BI И АРХИТЕКТУРА МОДЕЛИ ДАННЫХ (СХЕМА «ЗВЕЗДА»)

## 1. Введение в архитектуру: Что есть BI в конвейере данных
В СУБД мы храним данные в виде нормализованных таблиц, в Python — чистим и трансформируем массивы. Но для конечного потребителя (генерального директора, инвестора, начальника управления) миллионы строк в базе — это слепое пятно. 

**BI (Business Intelligence)** — это финальный завод по сборке готовой продукции. Он берет плоские таблицы, поднимает их в оперативную память и превращает в интерактивный пульт управления бизнесом (Дашборд).

* **В Excel:** Чтобы обновить отчет, ты вручную пересчитываешь ВПР, тянешь формулы, перестраиваешь сводные таблицы. При объеме более 500 000 строк файл «умирает».
* **В BI:** Ты один раз проектируешь жесткую архитектуру связей и формул. Данные обновляются по одной кнопке, пролетая через RAM за доли секунды, даже если на входе 10 000 000 строк.

---

## 2. Локальное развертывание поля боя (Инструкция для твоего ПК)
Для автономного изучения без платных корпоративных лицензий и облачных учетных записей мы используем **Microsoft Power BI Desktop** — мировой промышленный стандарт BI-аналитики. Программа полностью бесплатна для локального использования на одном компьютере.

### Пошаговый алгоритм установки и запуска:
1. Скачай **Power BI Desktop** с официального сайта Microsoft или через Microsoft Store на Windows.
2. Установи и запусти программу. Окно входа/регистрации просто закрывай (крестиком в углу) — для локальной работы профиль не нужен.
3. Перед тобой чистый холст. Сразу зайди в настройки, чтобы отключить ИИ-глянец, который ломает логику моделей:
   * Нажми `Файл` (File) → `Параметры и настройки` (Options and settings) → `Параметры` (Options).
   * В левом меню выбери `Загрузка данных` (Data Load) в блоке **Текущий файл** (Current File).
   * **Критически важно:** Сними галочку с пункта *«Автоматически обнаруживать новые отношения после загрузки данных»* и переключи «Автоматическая дата/время» в статус **Выключено**. Если этого не сделать, BI начнет самовольно плодить невидимые связи и скрытые таблицы дат, забивая RAM и ломая твои расчеты.

## 3. Инструмент №1: Архитектура связей — Схема «Звезда» (Star Schema)

### Логика инструмента
В BI нельзя просто свалить все данные в одну гигантскую «широкую» таблицу, как в кустарном Excel. Это приведет к дикому дублированию текста и параличу вычислений. BI требует жесткого разделения таблиц на два лагеря: **Таблицы Фактов (Fact Tables)** и **Таблицы Измерений / Справочники (Dimension Tables)**.

### Синтаксис и Структура Схемы
Архитектура строится по принципу «Звезды»: в центре находится тяжелая таблица фактов, а вокруг нее радиально располагаются легкие справочники. Они соединяются линиями связи по уникальным ключам (ID).

```text
       ┌────────────────────────┐
       │   dim_materials        │ (Справочник материалов)
       │   [material_id] (PK)   │
       └───────────┬────────────┘
                   │ 1
                   │
                   │ * 
 ┌─────────────────▼──────────────────┐       ┌────────────────────────┐
 │   fact_smr_delivery                │       │   dim_objects          │
 │   [delivery_id]                    │*     1│   [object_id] (PK)     │
 │   [object_id]   (FK) ──────────────┼───────►   [object_name]        │
 │   [material_id] (FK)               │       └────────────────────────┘
 │   [tons_fact]                      │
 └─────────────────▲──────────────────┘
                   │ *
                   │
                   │ 1
       ┌───────────┴────────────┐
       │   dim_calendar         │ (Справочник времени / Календарь)
       │   [date_id] (PK)       │
       └────────────────────────┘
```

1. **Таблица Фактов (`fact_...`):** Хранит непрерывный поток числовых логов (транзакций). Что произошло, сколько и когда. Содержит только цифры (метрики) и технические ключи-указатели (`ID`).
   * *Пример:* `fact_smr_delivery` (Колонки: `date_key`, `object_id`, `material_id`, `tons_fact`, `cost_rub`). Здесь нет текста «ЖК Центр», здесь лежит только числовой `object_id = 105`.
2. **Таблицы Измерений / Справочники (`dim_...`):** Хранят уникальные текстовые характеристики объектов. Отвечают на вопросы: *Кто? Что? Какой? Какая марка?*.
   * *Пример:* `dim_objects` (Колонки: `object_id`, `object_name`, `region`, `supervisor_name`). Каждому `object_id` соответствует строго одна уникальная строка с текстовым описанием.

## 4. Физика работы: Однонаправленная фильтрация и Кратность связей

Когда ты нажимаешь кнопку на дашборде (например, выбираешь регион «Саратов» в справочнике `dim_objects`), внутри BI активируется физический конвейер:
1. Фильтр намертво сжимает таблицу справочника `dim_objects`, оставляя в ней только строки по Саратову.
2. Логический сигнал летит по линии связи от справочника к таблице фактов `fact_smr_delivery`.
3. Таблица фактов мгновенно отсекает все строки, кроме тех, у которых `object_id` совпадает с выжившими id из справочника (полный аналог операции `WHERE object_id IN (...)`).

### Кратность связи: `1-к-Многим` (`1-to-Many` / `1:*`)
Это единственный математически верный тип связи в BI. 
* Со стороны справочника всегда стоит цифра **`1`** (потому что в справочнике каждый ID уникален и встречается строго один раз — Первичный ключ / *Primary Key*).
* Со стороны таблицы фактов всегда стоит звездочка **`*`** (потому что один и тот же объект или материал может привозиться на стройку многократно, генерируя множество строк — Внешний ключ / *Foreign Key*).

## 5. Капкан №1: Взрыв модели при связи «Многие-ко-Многим» (`Many-to-Many`)
Если ты попытаешься связать две таблицы напрямую по полю, которое дублируется в обеих таблицах (например, грязный текст названия материала без уникального ID), BI создаст связь `Many-to-Many` (`*:*`).

### Физика капкана
При связи `*:*` стрелка направления фильтрации становится двунаправленной. Сигнал фильтра начинает хаотично циркулировать между таблицами в обоих направлениях. Математический движок BI теряет точку опоры, начинает перемножать строки, задваивать или затраивать итоговые суммы (SUM), мгновенно забивает оперативную память RAM и выдает ложные экономические показатели.

### Защита Мастера
Никогда не связывай таблицы фактов напрямую друг с другом или по грязным текстовым полям. Если нужно связать две таблицы фактов (например, Смету и Факт поставок), ты обязан вытащить все уникальные материалы в отдельный изолированный справочник `dim_materials`, присвоить каждому строгий числовой `material_id` (сделать его уникальным `1`), и завязать обе таблицы фактов на этот справочник с кратностью `1:*`. Справочник выступит жестким мостом-барьером.

## 6. ПРАКТИЧЕСКИЙ БОЕВОЙ ПОЛИГОН (РАЗВЕРТЫВАНИЕ СОБСТВЕННЫМИ РУКАМИ)

Чтобы изучить физику «Звезды» без подключения к серверам, мы развернем модель из трех таблиц прямо внутри твоего файла Power BI Desktop через встроенный инструмент ручного ввода.

### Шаг 1. Заливка данных в RAM
1. На главной панели Power BI найди и нажми кнопку **«Ввести данные»** (Enter Data). Откроется пустое окно мини-Excel.
2. Переименуй таблицу внизу в `dim_objects`. Создай две колонки и вбей руками следующие строки:
   * `object_id` (целое число): `1`, `2`
   * `object_name` (текст): `ЖК Центр`, `Объект Восточный`
   * Нажми кнопку **«Загрузить»** (Load).
3. Снова нажми **«Ввести данные»**. Создай справочник материалов `dim_materials`:
   * `material_id` (число): `10`, `20`
   * `material_name` (текст): `Цемент М500`, `Арматура А500С`
   * Нажми **«Загрузить»**.
4. Снова нажми **«Ввести данные»**. Создаем тяжелую таблицу фактов `fact_delivery`:
   * `delivery_id` (число): `101`, `102`, `103`, `104`
   * `object_id` (число): `1`, `1`, `2`, `1`
   * `material_id` (число): `10`, `20`, `10`, `10`
   * `tons_fact` (число): `50.0`, `120.0`, `300.0`, `45.5`
   * Нажми **«Загрузить»**.

### Шаг 2. Инженерия связей (Сборка Звезды)
1. В крайнем левом вертикальном ряду иконок Power BI переключись на самую нижнюю вкладку — **«Модель»** (Model, иконка с тремя связанными квадратами). Перед тобой появятся три твои таблицы в виде блоков.
2. Расставь их глазами: `fact_delivery` по центру, справочники — по бокам.
3. Наведи мышку на поле `object_id` в таблице `dim_objects`, зажми левую кнопку мыши, потяни линию и брось её точно на поле `object_id` в центральной таблице `fact_delivery`. 
   * *Система создаст линию связи. Убедись, что у справочника стоит `1`, у фактов — `*`, а стрелочка указывает строго от справочника к фактам.*
4. Точно так же зажми поле `material_id` в `dim_materials` и протащи линию на `material_id` в `fact_delivery`.

# Глава 2. МАТЕМАТИЧЕСКИЙ ДВИЖОК DAX: ВЫЧИСЛЯЕМЫЕ МЕРЫ, КОНТЕКСТ СТРОКИ И КОНТЕКСТ ФИЛЬТРА

## 1. Введение в архитектуру: Как считает BI
В SQL мы пишем запросы на лету, а в Python используем векторизованные методы Pandas. Внутри Power BI бьется собственное математическое сердце — функциональный язык формул **DAX (Data Analysis Expressions)**. 

Его главная фича — умение производить сложнейшие финансовые и производственные расчеты в терабайтных массивах данных за миллисекунды, на лету адаптируясь к любым кнопкам (резакам), которые нажимает пользователь на дашборде.

**ГЛАВНЫЙ АРХИТЕКТУРНЫЙ ЗАКОН:** BI считает данные только тогда, когда на них смотрят. Формула DAX — это не жестко записанный результат, это **динамический алгоритм**, который ждет входящего сигнала от модели данных.

---

## 2. Инструмент №1: Вычисляемые Меры (Measures) против Вычисляемых столбцов (Calculated Columns)

### Логика инструмента
Новички, приходящие из Excel, совершают одну и ту же фатальную ошибку: они пытаются создавать новые столбцы с формулами внутри тяжелой таблицы фактов (например, умножают цену на количество в каждой строке). В BI так делать запрещено. Мы разделяем вычисления на **Меры** и **Столбцы**.

### Карта различий
1. **Вычисляемый столбец (Calculated Column):** Считается один раз при загрузке данных. Физически пишется на жесткий диск или в RAM для *каждой* строки. Пожирает память. Не умеет динамически реагировать на срезы пользователей.
2. **Вычисляемая Мера (Measure):** Настоящий экзоскелет аналитика. Физически занимает **0 байт** на диске. Это просто формула-инструкция, хранящаяся в модели. Она материализуется в RAM только в момент вывода на экран и рассчитывает итог мгновенно, подстраиваясь под контекст.

### Синтаксис создания Базовой Меры (Агрегаторы SUM / COUNT)
```dax
// Синтаксис: Имя_Меры = ФУНКЦИЯ( 'Имя_Таблицы'[Имя_Колонки] )

Total Tons = SUM( 'fact_delivery'[tons_fact] )

Total Deliveries = COUNT( 'fact_delivery'[delivery_id] )
```

---

## 3. Физика работы: Контекст строки против Контекста фильтра

Понимание DAX держится на двух фундаментальных понятиях. Ошибешься здесь — весь дашборд будет врать.

1. **Контекст строки (Row Context):** Существует, когда система идет построчно (как цикл `for` в Python). Например, когда мы внутри вычисляемого столбца пишем `[Цена] * [Количество]`, система знает, что на Строке №1 надо взять цену первой строки и количество первой строки. Меры по умолчанию контекста строки **не имеют** — они видят всю колонку целиком.
2. **Контекст фильтра (Filter Context):** Самое важное. Это окружение, в котором в данный момент работает Мера. Если ты вывел Меру `Total Tons` в карточку на дашборде, а рядом нажал срез «Цемент М500», Контекст фильтра сожмет таблицу фактов до цемента, и Мера выдаст сумму только по нему. Если срезов нет — Мера считает сумму по всей таблице.

---

## 4. Инструмент №2: Тяжелая артиллерия и Переключатель Контекста — Функция `CALCULATE()`

### Логика инструмента
`CALCULATE()` — это самый главный и мощный оператор в языке DAX. Это **абсолютный властелин контекста**. Она умеет брать текущий контекст фильтра (все кнопки, нажатые пользователем), хирургически вскрывать его, изменять, дописывать новые условия или полностью игнорировать внешние фильтры. 

Это прямой аналог конструкции `SUM(case when...)` или сложного подзапроса с жестким `WHERE` в SQL.

### Синтаксис
```dax
// Синтаксис: CALCULATE( [Имя_Существующей_Меры], Фильтр1, Фильтр2, ... )

Tons of Cement = CALCULATE( [Total Tons], 'dim_materials'[material_name] = "Цемент М500" )
```

### Физика работы
Когда BI натыкается на `CALCULATE()`, он ставит внешние фильтры на паузу, заходит внутрь функции, накладывает жесткое условие (в нашем примере — оставляет только «Цемент М500»), объединяет его с остальными внешними фильтрами и только потом запускает расчет базовой меры `[Total Tons]`.

## 5. Капкан №1: Попытка посчитать среднее через среднее столбца (`AVERAGE`)
Если тебе нужно посчитать среднюю цену за тонну по всему холдингу, и ты напишешь меру как `Average Price = AVERAGE('fact_delivery'[price_per_ton])`, ты попадешь в капкан ложной математики.

### Физика капкана
Функция `AVERAGE` просто сложит цены всех строк в таблице фактов и разделит на количество строк. Но одна строка могла быть поставкой 1 тонны, а другая — поставкой 500 тонн. Excel-аналитики часто совершают эту ошибку, выдавая «среднее по больнице», что приводит к искажению финансовых отчетов на миллионы рублей.

### Защита Мастера — Использование Итераторов (`SUMX`)
Чтобы посчитать экономически верный показатель, нужно использовать итераторы семейства `X` (например, `SUMX`, `AVERAGEX`). Они принудительно создают Контекст Строки для Меры. Они идут построчно, делают локальную математику, складывают итог в RAM и выдают верный результат.
```dax
// Защита: Сначала считаем общую стоимость всех строк, а потом делим на общий тоннаж
True Average Price = DIVIDE( SUMX( 'fact_delivery', 'fact_delivery'[tons_fact] * 'fact_delivery'[price_per_ton] ), [Total Tons] )
```
*Примечание:* Безопасный оператор `DIVIDE(Числитель, Знаменатель)` — это встроенный **защитный предохранитель от деления на ноль**. Если `[Total Tons]` равен 0, функция вернет пустоту (`BLANK`), не обрушив дашборд.

## 6. ПРАКТИЧЕСКИЙ БОЕВОЙ ПОЛИГОН (НАПИСАНИЕ МЕР РУКАМИ НА ПК)

Запускаем твой верстак Power BI Desktop, на котором в Главе 1 мы уже развернули и связали три таблицы схемы «Звезда». Наша задача — вдохнуть в модель математику.

### Шаг 1. Создание базовой Меры (SUM)
1. В крайнем левом ряду иконок переключись на самую верхнюю вкладку — **«Отчет»** (Report, иконка чистого холста).
2. В крайней правой панели **«Данные»** (Data) найди таблицу `fact_delivery`, нажми на нее правой кнопкой мыши и выбери **«Создать меру»** (New measure).
3. Наверху появится строка формул (как в Excel, но для кода). Удали всё, что там написано, и вставь жесткую формулу:
   ```dax
   Total Tons = SUM('fact_delivery'[tons_fact])
   ```
4. Нажми `Enter` или галочку слева от строки формул. В таблице `fact_delivery` появится иконка калькулятора с именем `Total Tons`.

### Шаг 2. Создание Меры Управления Контекстом (CALCULATE)
1. Снова нажми правой кнопкой мыши по `fact_delivery` → **«Создать меру»**.
2. Вбей формулу, которая будет жестко вырезать только объемы цемента, игнорируя текстовый хаос:
   ```dax
   Cement Only Tons = CALCULATE([Total Tons], 'dim_materials'[material_name] = "Цемент М500")
   ```
3. Нажми `Enter`.

### Шаг 3. Визуальный аудит работы движка в RAM
1. На панели **«Визуализации»** (Visualizations) найди иконку **«Карточка»** (Card, иконка с цифрами `1,2,3`). Нажми на нее. На холсте появится пустой квадрат.
2. Перетащи мышкой меру `Total Tons` из панели данных внутрь этого квадрата. Карточка мгновенно отобразит цифру: **515.5** (общая сумма всех тонн из нашей таблицы фактов).
3. Создай вторую **«Карточку»** и перетащи в нее меру `Cement Only Tons`. Она отобразит: **395.5** (система на лету отфильтровала строки, сложив только поставки Цемента для ЖК Центр и Объекта Восточного: 50.0 + 300.0 + 45.5).

# Глава 3. ПРОЕКТИРОВАНИЕ ВИТРИНЫ УПРАВЛЕНИЯ: ДИНАМИЧЕСКИЕ СРЕЗЫ И СВЕТОФОРНАЯ СИГНАЛИЗАЦИЯ (УСЛОВНОЕ ФОРМАТИРОВАНИЕ)

## 1. Введение в архитектуру: Дашборд как операционная панель контроля
Мы подошли к финальному этапу конвейера. Таблицы фактов увязаны со справочниками в схему «Звезда» (Глава 1), вычислительное ядро DAX готово пересчитывать тонны и кубы за миллисекунды (Глава 2). Теперь нам нужно материализовать визуальный интерфейс — **Витрину данных (Дашборд)**.

**ГЛАВНЫЙ АРХИТЕКТУРНЫЙ ЗАКОН:** Дашборд аналитика данных — это не выставка красивых картинок и абстрактных 3D-графиков. Это **приборная панель управления (Dashboard)**, как в кабине пилота или на щите управления АЭС. Её задача — за 2 секунды взгляда руководителя подсветить критические точки: *Где система идет штатно, а где горит аварийный сигнал перерасхода лимитов?*

---

## 2. Инструмент №1: Элементы управления потоком — Срезы (Slicers)

### Логика инструмента
Срез — это интерактивная визуальная кнопка-рубильник на холсте дашборда. Он вытаскивает уникальные текстовые значения из Справочников (`dim_...`) и при нажатии мгновенно формирует новый Контекст Фильтра для всего холста.

### Физика работы
Когда пользователь кликает на срезе по объекту «ЖК Центр», срез отправляет по линиям связи «Звезды» сигнал «Истины» (`True`) для `object_id = 1`. Все графики, таблицы и карточки на холсте моментально сжимают свои внутренние таблицы до этого ID и пересчитывают меры за доли секунды в RAM.

---

## 3. Инструмент №2: Светофорная зачистка — Условное форматирование (Conditional Formatting)

### Логика инструмента
Инструмент динамического окрашивания ячеек таблиц или графиков в зависимости от математического значения меры. Это твой главный инструмент борьбы с «информационной слепотой» руководства.

### Синтаксис логических правил (Цветовой код)
*   **Красный Сигнал (Авария):** Значение Меры вылетело за рамки бюджета или лимита (Перерасход). Требуется жесткое вмешательство ИТ/ПТО.
*   **Желтый Сигнал (Внимание):** Значение приближается к критической границе (Остаток лимита менее 10%).
*   **Зеленый Сигнал (Норма):** Система функционирует в штатном режиме, объемы в пределах сметы.

---

## 4. Капкан №1: Информационный хаос («Эффект Попугая»)
Самый частый капкан начинающих аналитиков — использование круговых диаграмм (Pie Charts) на 20 категорий, разноцветных фонов холста и десятков контрастных цветов в одной таблице.

### Физика капкана
Когда дашборд пестрит всеми цветами радуги без строгой логики, мозг руководителя испытывает когнитивную перегрузку. Взгляд рассеивается, фокус внимания теряется, и критическая аномалия (например, перерасход арматуры на 50 тонн) сливается с общим ярким фоном. Такой дашборд приносит бизнесу вред, маскируя баги управления.

### Защита Мастера — Правило Минимализма и Светофора
Весь твой дашборд должен быть выполнен в нейтральных, спокойных тонах (белый, светло-серый, темно-синий для заголовков). Яркие цвета — **Красный** и **Зеленый** — имеют право появляться на холсте **исключительно как маркеры условного форматирования**. Если ячейка горит красным, это должно означать только одно: «Здесь авария». Если перерасхода нет — ячейка остается нейтрально серой или белой.

## 5. ПРАКТИЧЕСКИЙ БОЕВОЙ ПОЛИГОН (СБОРКА ПУЛЬТА УПРАВЛЕНИЯ НА ПК)

Завершаем сборку твоего автономного BI-комплекса в Power BI Desktop. Нам нужно собрать визуальную таблицу контроля лимитов со светофорной сигнализацией, реагирующую на срезы.

### Шаг 1. Подготовка Сметной Меры контроля
Чтобы светофор понимал, где перерасход, создадим быструю меру лимита.
1. В панели «Данные» нажми правой кнопкой по `fact_delivery` → **«Создать меру»**.
2. Вбей формулу жесткого сметного лимита (для теста зафиксируем сметную планку в 200 тонн):
   ```dax
   Limit Tons = 200
   ```
3. Создай еще одну меру — индикатор перерасхода (Векторная разница):
   ```dax
   Over Limit = [Total Tons] - [Limit Tons]
   ```

### Шаг 2. Сборка Интерактивной Таблицы
1. На панели **«Визуализации»** (Visualizations) найди иконку **«Таблица»** (Table) и кликни на нее. На холсте появится пустой табличный блок.
2. Перетащи из панели данных внутрь этой таблицы следующие поля:
   * Из `dim_materials` \(\rightarrow\) `material_name`
   * Из `fact_delivery` \(\rightarrow\) меры `Total Tons`, `Limit Tons` и `Over Limit`.
3. Таблица мгновенно рассчитает строки по материалам холдинга.

### Шаг 3. Монтаж Рубильника (Среза)
1. Кликни по пустому месту на холсте дашборда (чтобы снять выделение с таблицы).
2. На панели визуализаций выбери иконку **«Срез»** (Slicer, выглядит как плитка с фильтром).
3. Перетащи в него поле `object_name` из справочника `dim_objects`. Срез превратится в список объектов с галочками.

### Шаг 4. Настройка Светофора (Защита Мастера)
1. Выдели твою созданную таблицу на холсте.
2. На панели визуализаций в блоке «Столбцы» (Columns) найди меру `Over Limit`, нажми на стрелочку рядом с ней и выбери **«Условное форматирование»** (Conditional formatting) \(\rightarrow\) **«Цвет фона»** (Background color).
3. В открывшемся окне настроек:
   * Выберите стиль формата: **«Правила»** (Rules).
   * Базовое поле, на котором строится логика: `Over Limit`.
   * Настрой правило: Если значение **больше 0** (Идет перерасход), то окрасить ячейку в **Красный цвет**. Если значение **меньше или равно 0** (Мы в лимите), окрасить в **Нейтральный серый / Белый цвет**.
   * Нажми `ОК`.

### Контрольный запуск всей системы в RAM:
Посмотри на холст дашборда. Сейчас в таблице мера `Over Limit` для Арматуры показывает перерасход, и ячейка горит **красным аварийным цветом**. 
Теперь подойди к срезу объектов и нажми галочку на **«Объект Восточный»**. 

Сигнал пролетает по схеме «Звезда», сужает контекст фильтра до ID=2. Таблица фактов пересчитывается в RAM, объемы арматуры на этом объекте падают в ноль, мера `Over Limit` уходит в минус, и красный цвет с холста **мгновенно исчезает**, возвращая систему в штатный статус контроля.
